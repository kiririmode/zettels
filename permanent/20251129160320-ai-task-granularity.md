## 要約（Summary）

- 単一APIの実装と比較して、複数API（3〜4つ）をまとめて実装すると、1APIあたりの作業時間が約66%削減される
- これは、AIへのドメイン知識学習コストが複数タスクで償却されるため
- AIの性能を最大化するには、タスクを適切な粒度で「まとめ、任せる」ことが重要

## 本文（Body）

自律型AIを活用する際、タスクの粒度が生産性に大きく影響する。個別タスクではなく、まとめて委譲することで劇的な効率化が可能になる。

### 背景・問題意識

自律型AIは強力だが、以下のような「準備コスト」がかかる：
- ドメイン知識の共有（テーブル定義、ビジネスルール等）
- コーディング規約の説明
- 設計思想の明文化

これらを1タスクごとに行うのは非効率的である。

### アイデア・主張

**複数の関連タスクをまとめてAIに委譲することで、準備コストを償却し、1タスクあたりの効率を大幅に向上できる**。

#### 効率化のメカニズム

```
単一API実装の場合:
準備コスト（100分） + 実装時間（50分） = 150分/API

複数API（4つ）実装の場合:
準備コスト（100分） + 実装時間（50分×4） = 300分 → 75分/API（50%削減）

実測では約66%の削減効果が確認された。
```

#### 最適なタスク粒度

- **小さすぎる**: 準備コストが毎回発生し、非効率
- **大きすぎる**: 複雑になりすぎて、AIがエラーを起こしやすくなる
- **最適**: 3〜4つの関連APIをまとめる程度

### 具体例・ケース

食べログのAPI開発での実践：

**Before（個別実装）**:
- 予約作成API: 準備100分 + 実装50分 = 150分
- 予約更新API: 準備100分 + 実装50分 = 150分
- 予約キャンセルAPI: 準備100分 + 実装50分 = 150分
- **合計: 450分**

**After（一括実装）**:
- 予約関連API（作成・更新・キャンセル）: 準備100分 + 実装150分 = 250分
- **削減率: 約44%**

さらに、AIが学習したドメイン知識を次の関連タスクでも再利用できるため、継続的に効率が向上する。

### 反論・限界・条件

- 関連性の低いタスクをまとめると、逆に混乱を招く
- 初回は準備コストが高いため、単発タスクには向かない
- AIの能力によっては、大きなタスクを処理しきれない場合がある
- タスクをまとめることで、エラー発生時のデバッグが複雑になる可能性
